================================================================================
                    FACTORY WHATSAPP AI AUTOMATION BOT
================================================================================

Overview:
This application is a WhatsApp automation bot that acts as an intelligent bridge
between a factory WhatsApp group and a Google Sheet. It listens to a specific 
group, processes incoming requests (both written text and voice notes), extracts
key details of requested machinery parts using the Gemini 2.5 Flash API, and 
appends the structured details to a Google Sheet.

Step-by-Step Flow:
------------------
1. Connection & Session:
   - The bot initializes the WhatsApp Client using 'whatsapp-web.js' with 'LocalAuth'
     to persist the session.
   - If a new login is required, it outputs a QR code in the terminal via 'qrcode-terminal'.
   - Once connected, it logs "WhatsApp Connected".

2. Listening & Filtering:
   - The bot listens to all incoming messages.
   - It retrieves the chat object and filters messages to process ONLY those coming from the 
     TARGET_GROUP ("120363427181556541@g.us"). All other messages are ignored.

3. Message Processing:
   - A. Text Messages (Plain message with no media):
     - Logs the text message and forwards it to `processText(text)`.
   - B. Voice Notes (Audio media):
     - Downloads the audio file from WhatsApp.
     - Saves the file locally with a temporary filename `voice_<timestamp>.ogg`.
     - Converts the binary file into a base64 string and forwards it to `processAudio(filename)`.

4. AI Information Extraction (Gemini):
   - The bot instantiates the Gemini API using the `gemini-2.5-flash` model.
   - The text or audio prompt instructs Gemini to:
     - Translate multilingual messages (e.g., Hindi, Punjabi, etc.) to English.
     - Parse the input, which might request one or multiple items.
     - For EACH item, extract the following 6 fields:
       * Part Name
       * Qty Required (extracted as a plain number e.g., "1", "20")
       * Size (empty if not mentioned)
       * Material (empty if not mentioned)
       * For Machine (empty if not mentioned)
       * Vendor (empty if not mentioned)
     - Propagate variables: If "Vendor" or "For Machine" is mentioned once for multiple items,
       the prompt instructs Gemini to apply it to ALL items in that message.
     - Output ONLY a raw JSON array containing objects matching the schema.

5. JSON Parsing & Resiliency:
   - Cleans the raw response from Gemini (strips ```json markdown wrappers).
   - Attempts to parse the response into a JSON array.
   - If parsing fails, it falls back to a default JSON format placing the raw output in 
     the "Part Name" field to avoid losing data.

6. Pending Queue Redirection:
   - Forwards each extracted part item to the dashboard Express server pending queue (`POST /api/pending`).
   - Maps Gemini's extracted keys to standard camelCase properties:
     * `partName` (derived from "Part Name")
     * `qty` (derived from "Qty Required")
     * `size` (derived from "Size")
     * `material` (derived from "Material")
     * `machine` (derived from "For Machine")
     * `vendor` (derived from "Vendor")
   - This holds the items in an intermediate queue awaiting admin verification before writing to Google Sheets.

5. TECHNICAL RECOMMENDATIONS & PLANNED IMPROVEMENTS
--------------------------------------------------------------------------------

1. Temp File Clean-Up:
   - When a voice note is received, `fs.writeFileSync(filename, ...)` is called.
   - These `.ogg` files accumulate in the root workspace directory.
   - Improvement: Use `fs.unlinkSync(filename)` in a `finally` block or after Gemini processing 
     to clean up temporary files automatically (similar to `test2.js`).

2. Robust JSON Extraction:
   - Even though Gemini is instructed to return only a raw JSON array, models sometimes output 
     markdown fences (e.g. ```json ... ```) or prefix text. 
   - Improvement: Utilize Gemini's Structured Outputs feature (using `responseSchema` in the 
     generation configuration) to guarantee 100% syntactically correct JSON array responses.

3. Process Audio directly from Buffer:
   - Instead of writing to disk, you can pass the base64 buffer directly from WhatsApp media.data
     without saving a file, which improves I/O performance and eliminates local file management.

6. WEB DASHBOARD INTEGRATION (REACT & EXPRESS)
--------------------------------------------------------------------------------

A light, minimalist web dashboard inspired by high-end design languages (e.g., the SPARK layout) is integrated into the project for visualizing part requests in real-time.

Architecture:
-------------
- Backend: Built with Express.js (`server.js`) listening on Port 3001. It uses a Postgres database (Neon DB) to store unapproved requests received from the bot. It exposes:
  * `GET /api/requests`: Reads approved rows directly from the Google Sheet.
  * `GET /api/pending`: Reads the Neon DB pending queue (persisted in `pending_requests`).
  * `POST /api/pending`: Adds a new parsed demand to the database.
  * `POST /api/pending/:id/approve`: Saves/Appends approved details to the Google Sheet and updates status to `approved` in DB.
  * `POST /api/pending/:id/reject`: Rejects the item in the database (status = `rejected`).
  * `POST /api/pending/:id/edit`: Edits and updates the request in the database.
  * `POST /api/pending/:id/forward`: Forwards the request from Editor to Manager (status = `reviewed`).
  * Static routes: Serves static voice files from `/audio` and static React built files.
- Frontend: Built with React, Vite, and Vanilla CSS, located in the `dashboard/` directory.

Visual Theme & Layout:
----------------------
- White/Slate minimalist: High whitespace, crisp `#ffffff` sidebar, soft `#f8fafc` background.
- Sidebar Tabs: Includes toggleable tabs for "Pending Demands" (active by default), "Receiving Queue", "Approved History", and "Inventory Catalog".
- Dynamic Notification Badge: The pending sidebar button displays a red numerical indicator detailing the current size of the queue for the active role.
- KPI Statistic Cards: The Approver Dashboard features a top-level KPI Summary Panel displaying real-time statistis fetched dynamically from Neon DB:
  * *New Worker Demands* (`status = 'pending_review'`)
  * *Pending Approval* (`status = 'reviewed'`)
  * *Approved but Not Received* (`status = 'approved'`)

Worker → Editor → Approver Workflow:
------------------------------------
- The dashboard enforces a strict sequential role hierarchy based on bcrypt-authenticated profiles (`users.json`):
  * **Editor** (Reviewer): Sees only incoming worker requests (`status = 'pending_review'`). Actions include Edit, Forward for Approval, and Reject.
  * **Approver** (Manager): Sees only requests reviewed and forwarded by the Editor (`status = 'reviewed'`). Actions include Edit, Approve, and Reject.
- Display mode (Card View vs List View) does not affect actions or permissions; identical action blocks are accessible in both views.

Procurement-Flexible Searchable Autocomplete:
---------------------------------------------
- **9 Consistent Fields**: Both Card View and List View edit forms expose the exact same 9 fields: Part Name, SKU (P No.), Reg No., Quantity, Size specs, Material, Machine, Vendor, and Rate.
- **Searchable Autocomplete Input**: Core procurement fields (Part Name, Vendor, Machine, Material) use standard `<input type="text">` fields bound to `<datalist>` autocomplete options fetched from the live inventory catalog.
- **Procurement Flexibility**: Unlike strict select dropdowns, datalist-assisted inputs present useful catalog recommendations while allowing the Editor to freely type and save brand new custom procurement values (e.g. new parts or vendors) that don't already exist in the inventory.
- **Dynamic Auto-Fill Relationship**: Selecting an item from the Part Name, SKU, or Reg No. lists automatically queries the preloaded master array and auto-fills all other fields in the form (Size, Material, Machine, Vendor, Rate, SKU) to prevent typing errors.

7. INVENTORY CATALOG PRICING & RATE INTEGRATION
--------------------------------------------------------------------------------
- **Inventory Rate Column**: A new `rate` (NUMERIC(10, 2) DEFAULT 0.00) column is added to the Neon DB `inventory` catalog table.
- **Copying Specs upon Approval**: When a new demand request is approved by the Manager (creating a *new* inventory item), the system automatically copies the following fields from the approved request into the new inventory catalog record: Part Name, P No. (sku), Reg No., Vendor, Rate, Material, Size, and Machine.
- **Preserve Catalog Pricing**: If the inventory item already exists in the catalog, approval does *not* automatically overwrite the catalog rate, preserving the master pricing.
- **Separate Rate Fields**: The request rate (stores the negotiated price used for that specific approval) and the inventory rate (stores the catalog master list price) remain completely separate.
- **Double Layout Support (Card & List Views)**: The rate is prominently displayed as currency in both:
  * *Inventory List (Table) View*: Rendered as a distinct "Rate" column.
  * *Inventory Card Grid View*: Rendered beautifully inside the inventory items' specification lists.
